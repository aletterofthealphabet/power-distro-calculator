# Power Distro Calculator — Design Document

Concrete architecture, data model, file layout, and key function
signatures, derived from `IMPLEMENTATION_PLAN.md` (domain model §3,
calc engine §4, stack §5, roadmap §6, defaults §7a, draft schema §8,
API sketch §9). This document is the thing Phase 0 scaffolding is
built against — where the plan says "what and why," this says
"exactly which files and functions."

Stack (per plan §5/§7a.1): Node.js + TypeScript monorepo, React +
Vite frontend, Fastify backend, Prisma + PostgreSQL, pure-TS calc
engine shared by both.

## 1. Architecture

### 1.1 System shape

```
┌──────────────────┐        ┌──────────────────┐        ┌──────────────┐
│   apps/web        │  HTTP  │   apps/api        │  SQL   │  PostgreSQL  │
│  React + Vite      │──────▶│  Fastify + Prisma  │──────▶│              │
│                    │◀──────│                    │◀──────│              │
└─────────┬──────────┘        └─────────┬──────────┘        └──────────────┘
          │                             │
          │   import (no network)       │   import (no network)
          ▼                             ▼
        ┌───────────────────────────────────┐
        │      packages/calc-engine           │
        │  pure TS, zero deps, zero I/O        │
        └───────────────────────────────────┘
                       ▲
                       │ shared types
        ┌───────────────────────────────────┐
        │      packages/shared-types          │
        └───────────────────────────────────┘
```

- **`calc-engine` is imported directly by both `web` and `api`** — not
  called over HTTP from the frontend. The frontend gets instant
  client-side feedback while a plot is being edited; the API re-runs
  the same engine server-side as the source of truth whenever a plot
  is saved or `/analyze` is called. Same code, same answer, no drift.
- **The API is the only thing that talks to Postgres.** The frontend
  never queries the DB directly (no direct Prisma-from-browser).
- **One-way data flow for optimization:** `auto-balance` computes a
  *proposed* diff; nothing is written to the DB until the user
  confirms via a normal `PATCH` (plan §9). The optimizer is a pure
  function over data already loaded — it doesn't reach into Prisma.

### 1.2 Why this split
- Calc engine isolated from DB/HTTP because it's the
  safety-critical, highest-test-coverage code (plan §4 preamble) —
  isolating it means it can be fuzz-/property-tested with plain data
  in, data out, no fixtures or mocking required.
- `shared-types` isolated from `calc-engine` because the engine
  should have **zero** dependencies (not even on Prisma-generated
  types) — DB-shaped types (with `id`, `createdAt`, etc.) live in
  `shared-types` or are generated by Prisma; the engine takes plain
  input DTOs mapped from those at the API/web boundary.

## 3. Data model

### 3.1 Prisma schema (refines plan §8's raw SQL into the actual
Phase-2 artifact — types/relations only, exact constraints finalized
during migration authoring)

```prisma
// apps/api/prisma/schema.prisma

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  createdAt DateTime @default(now())

  plots     Plot[]
  specs     EquipmentSpec[] @relation("SpecAuthor")
}

model EquipmentSpec {
  id               String   @id @default(uuid())
  name             String
  category         String
  powerWatts       Decimal?
  currentAmps      Decimal?
  voltage          Decimal
  powerFactor      Decimal?
  phase            Int      // 1 | 3
  connectorType    String
  isContinuousLoad Boolean  @default(true)
  notes            String?
  source           String?
  unverified       Boolean  @default(false)
  createdById      String?
  createdBy        User?    @relation("SpecAuthor", fields: [createdById], references: [id])
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  instances        EquipmentInstance[]
}

model CableSpec {
  id                      String   @id @default(uuid())
  gaugeAwg                String
  conductorCount          Int
  connectorType           String
  ratedAmps               Decimal
  resistanceOhmsPer1000ft Decimal
  source                  String?
  unverified              Boolean  @default(false)

  instances               EquipmentInstance[]
}

model Plot {
  id        String   @id @default(uuid())
  name      String
  venue     String?
  eventDate DateTime?
  ownerId   String?
  owner     User?    @relation(fields: [ownerId], references: [id])
  createdAt DateTime @default(now())

  distroUnits DistroUnit[]
  instances   EquipmentInstance[]
}

model DistroUnit {
  id             String  @id @default(uuid())
  plotId         String
  plot           Plot    @relation(fields: [plotId], references: [id], onDelete: Cascade)
  name           String
  inputConnector String
  maxAmps        Decimal
  phaseConfig    Int     // 1 | 3
  voltage        Decimal

  circuits       Circuit[]
}

model Circuit {
  id                    String   @id @default(uuid())
  distroUnitId          String
  distroUnit            DistroUnit @relation(fields: [distroUnitId], references: [id], onDelete: Cascade)
  breakerRatingAmps      Decimal
  voltage                Decimal
  phaseLeg               String   // 'L1' | 'L2' | 'L3' | 'hot' | 'neutral'
  connectorType           String
  isContinuousOverride    Boolean?

  instances               EquipmentInstance[]
}

model EquipmentInstance {
  id              String   @id @default(uuid())
  plotId          String
  plot            Plot     @relation(fields: [plotId], references: [id], onDelete: Cascade)
  equipmentSpecId String
  equipmentSpec   EquipmentSpec @relation(fields: [equipmentSpecId], references: [id])
  circuitId       String?
  circuit         Circuit?  @relation(fields: [circuitId], references: [id])
  cableSpecId     String?
  cableSpec       CableSpec? @relation(fields: [cableSpecId], references: [id])
  cableLengthFt   Decimal?
  quantity        Int      @default(1)
  pinned          Boolean  @default(false)
}
```

Relationships match plan §3 exactly:
`Plot 1—N DistroUnit 1—N Circuit 1—N EquipmentInstance N—1 EquipmentSpec`,
with `EquipmentInstance` optionally owning a `CableSpec` + `cableLengthFt`.

### 3.2 calc-engine input/output types (`packages/shared-types/src/analysis.ts`)

The engine never touches Prisma models directly — the API/web layers
map DB rows into these plain DTOs at the boundary.

```ts
// units.ts (calc-engine) — branded primitives to prevent unit mix-ups
export type Amps = number & { readonly __unit: 'A' };
export type Volts = number & { readonly __unit: 'V' };
export type Watts = number & { readonly __unit: 'W' };
export type Ohms  = number & { readonly __unit: 'ohm' };

// shared-types/analysis.ts
export interface EquipmentLoadInput {
  instanceId: string;
  quantity: number;
  powerWatts?: number;
  currentAmps?: number;
  voltage: number;
  isContinuousLoad: boolean;
  circuitId: string | null;
  pinned: boolean;
}

export interface CircuitInput {
  circuitId: string;
  breakerRatingAmps: number;
  voltage: number;
  phaseLeg: 'L1' | 'L2' | 'L3' | 'hot' | 'neutral';
  connectorType: string;
  connectorMaxAmps: number;      // resolved from a connector-type table
  isContinuousOverride?: boolean;
  cable?: { ratedAmps: number; resistanceOhmsPer1000ft: number; lengthFt: number };
}

export interface DistroInput {
  distroUnitId: string;
  maxAmps: number;
  phaseConfig: 1 | 3;
  circuits: CircuitInput[];
}

export interface PlotInput {
  plotId: string;
  distros: DistroInput[];
  equipment: EquipmentLoadInput[];
  voltageDropThresholdPct: number; // configurable, plan §4.2 / §7a.2
}

export type Severity = 'ok' | 'warning' | 'violation';

export interface Violation {
  severity: Severity;
  checkId:
    | 'continuous_derating' | 'cable_ampacity' | 'connector_rating'
    | 'distro_capacity' | 'voltage_drop' | 'phase_leg_overcurrent';
  circuitId?: string;
  distroUnitId?: string;
  message: string;         // human-readable explanation
  necReference?: string;   // optional citation, per §7a.2
}

export interface CircuitResult {
  circuitId: string;
  loadAmps: number;
  loadPct: number;         // vs. effective (derated) capacity
}

export interface PhaseLegResult {
  distroUnitId: string;
  leg: string;
  totalAmps: number;
}

export interface BalanceResult {
  legs: PhaseLegResult[];
  deviationPct: number;    // (max-min)/avg across legs
}

export interface AnalysisReport {
  circuits: CircuitResult[];
  balance: BalanceResult;
  violations: Violation[];
  totalDrawAmps: number;
}

export interface ReassignmentProposal {
  instanceId: string;
  fromCircuitId: string | null;
  toCircuitId: string;
}

export interface BalanceProposal {
  proposed: ReassignmentProposal[];
  projectedBalance: BalanceResult;
}
```

## 4. Key functions

All signatures below are pure (no I/O) unless in `apps/api/src` or
`apps/web/src`, matching the architecture split in §1.

### 4.1 `packages/calc-engine/src/load.ts`
```ts
function computeItemAmps(item: EquipmentLoadInput): Amps;
// amps = currentAmps if given, else powerWatts / voltage (plan §4.1)

function aggregateCircuitLoad(
  circuit: CircuitInput,
  equipment: EquipmentLoadInput[],
): Amps;
// Σ computeItemAmps(item) * item.quantity for items assigned to this circuit
```

### 4.2 `packages/calc-engine/src/checks/*.ts` — one function per check,
each returning `Violation[]` (empty if passing), never throwing:
```ts
function checkContinuousDerating(circuit: CircuitInput, loadAmps: Amps): Violation[];
// loadAmps <= 0.8 * breakerRatingAmps if continuous, else <= breakerRatingAmps

function checkCableAmpacity(circuit: CircuitInput, loadAmps: Amps): Violation[];
// loadAmps <= cable.ratedAmps AND cable.ratedAmps >= breakerRatingAmps

function checkConnectorRating(circuit: CircuitInput, loadAmps: Amps): Violation[];

function checkDistroCapacity(distro: DistroInput, circuitLoads: CircuitResult[]): Violation[];
// Σ circuit loads on this distro <= distro.maxAmps

function computeVoltageDrop(circuit: CircuitInput, loadAmps: Amps): number; // % drop
function checkVoltageDrop(circuit: CircuitInput, loadAmps: Amps, thresholdPct: number): Violation[];

function checkPhaseLegOvercurrent(legTotals: PhaseLegResult[], distro: DistroInput): Violation[];
```

### 4.3 `packages/calc-engine/src/balance/greedyLpt.ts`
```ts
function balanceGreedyLpt(
  distro: DistroInput,
  equipment: EquipmentLoadInput[],
): BalanceProposal;
// 1. Sort unpinned items by amps descending (Longest Processing Time).
// 2. Assign each to the currently-lightest eligible leg (only legs
//    reachable via the item's existing circuit/cable topology —
//    plan §4.3 "cabling-aware constraint").
// 3. Pinned items keep their current circuit and count toward leg totals
//    but are never reassigned.
// Returns a *proposal* — caller decides whether to apply it (plan §9).
```

### 4.4 `packages/calc-engine/src/analyze.ts` — the orchestrator, this is
the one function both `apps/api` and `apps/web` actually call:
```ts
function analyzePlot(input: PlotInput): AnalysisReport;
// For each distro → for each circuit: aggregateCircuitLoad, run all
// checks in 4.2, collect violations. Aggregate phase-leg totals per
// distro, compute BalanceResult.deviationPct. Sum totalDrawAmps.
// Deterministic, no randomness, no I/O — safe to call on every
// keystroke client-side (plan §6 Phase 4) and again server-side on save.
```

### 4.5 `apps/api/src/services/plotLoader.ts`
```ts
async function loadPlotInput(prisma: PrismaClient, plotId: string): Promise<PlotInput>;
// Fetches Plot + nested distroUnits/circuits/equipmentInstances via
// Prisma, maps DB rows -> calc-engine's plain PlotInput DTOs (§3.2).
// This is the ONLY place Prisma models are translated into engine input.
```

### 4.6 `apps/api/src/services/analysisService.ts`
```ts
async function analyze(prisma: PrismaClient, plotId: string): Promise<AnalysisReport>;
// loadPlotInput -> analyzePlot. Backs POST /plots/:id/analyze.

async function autoBalance(prisma: PrismaClient, plotId: string): Promise<BalanceProposal>;
// loadPlotInput -> balanceGreedyLpt per distro, merged. Backs
// POST /plots/:id/auto-balance. Does NOT write to the DB — plan §9's
// "propose, then confirm via PATCH" flow.
```

### 4.7 `apps/web/src/hooks/useLiveAnalysis.ts`
```ts
function useLiveAnalysis(plotDraft: PlotInput): AnalysisReport;
// Calls analyzePlot() directly (imported from calc-engine, no network
// round trip) on every plotDraft change, debounced to animation frame,
// for instant feedback while building a plot (plan §6 Phase 4).
```

## 5. Notes carried over from the plan (not re-litigated here)

- Regulatory thresholds (derating %, voltage-drop %) are configurable
  inputs to the engine, not hardcoded — §7a.2. `necReference` on a
  `Violation` is annotation-only.
- Ampacity/resistance tables (`packages/calc-engine/src/tables/`) are
  seeded from NEC 310.16-derived figures with a `source` field and an
  `unverified` flag per §7a.4 — do not treat unverified rows as
  authoritative until cross-checked.
- Local-search/ILP rebalancing (`balance/localSearch.ts`) is stubbed
  as an unimplemented placeholder per §7a.5 — out of scope until
  greedy LPT proves insufficient on real plot sizes.
- This document doesn't reopen §7's open questions; it assumes the
  §7a defaults hold until a human overrides them.
